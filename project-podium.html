Podium

Podium is a web application built for Magic: The Gathering Commander playgroups who want structured, competitive tracking of their multiplayer sessions. It replaces spreadsheets and scattered notes with a clean, modern system for logging games, analyzing performance, and visualizing long-term progression.

Details

Description

Podium is a data-driven MTG Commander tracking platform designed around real multiplayer game nights. Users can create pods, log sessions, record decks and commanders played, and track wins across players and groups. Over time, the app aggregates meaningful statistics such as win rates, head-to-head records, pod-size performance, and deck efficiency. Built with a focus on clarity and usability, Podium transforms casual match history into transparent, competitive insight. The interface is designed for dark-mode environments and structured dashboards, ensuring results remain readable, scalable, and easy to analyze across long-term playgroups.

Challenges

Learning React While Building at Scale
Podium was my first large-scale project built with React. Instead of experimenting with small demos, I committed to building a production-level application with complex state management, dynamic forms, conditional rendering, and reusable components. Structuring the app properly from the beginning while still learning React patterns was a significant challenge.

Managing a Large, Evolving Codebase
As the project grew to include sessions, groups, player identities, deck imports, statistics, and dashboards, maintaining consistency across components became increasingly complex. Refactoring early architectural decisions without breaking functionality required careful planning and iterative restructuring.

Firebase Security & Permission Architecture
Implementing Firestore security rules and user-based permissions proved to be one of the most technically demanding aspects. Ensuring that users could only access their groups, players could claim identities securely, session data remained protected, and queries did not break due to rule restrictions required multiple revisions of the database structure and rule logic. Debugging permission-denied errors in production-like scenarios forced a deeper understanding of Firebase’s security model.

Insights

Build Structure Early, Even While Learning
Jumping straight into a full application accelerated my learning far more than isolated tutorials. Designing with scalability in mind from the beginning saved significant refactoring later.

Security Rules Shape Architecture
Firestore permissions are not an afterthought. They directly influence how data should be structured. I learned to design collections and document relationships around security constraints, not just convenience.

Data Becomes a Product Feature
Tracking multiplayer Commander sessions revealed how raw data transforms casual gameplay into measurable progression. Designing meaningful analytics such as win rates, head-to-head stats, and pod performance required thinking beyond basic CRUD and into product strategy.

Iteration is Design
UI and branding went through multiple redesigns. The biggest insight was that identity, clarity, and usability evolve together. A strong product feel is rarely achieved in the first version.
